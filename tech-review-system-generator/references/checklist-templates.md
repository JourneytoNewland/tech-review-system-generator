# 检查项生成模板

不同技术类型的检查项生成规则和模板。

## 使用说明

根据评审项的技术类型，选择对应的模板生成检查项。

每个评审项包含 3-6 个检查项，覆盖核心技术点。

---

## 架构设计类

### 模板

```
- 设计完整性：核心组件/模块定义完整
- 设计合理性：技术选型有充分依据
- 扩展性：支持未来功能扩展
- 一致性：与其他模块协调一致
- 文档完备性：设计文档完整清晰
- 安全性：安全风险有应对方案
```

### 应用场景

- 系统架构设计
- 技术选型决策
- 模块划分设计
- 接口定义设计

## 疑问澄清类 🔍

### 特殊说明

此类评审项用于标记大模型在理解技术方案时发现的**逻辑疑问点**、**信息缺失**或**需要澄清的内容**。需要方案作者提供补充说明或修正。

### 模板

```
- 问题点：清晰描述疑问点或不清楚的地方
- 缺失信息：列出缺失的关键信息
- 影响范围：说明该疑问点影响的设计或实现范围
- 建议方案：提供可能的解决方案方向
- 需要澄清的内容：明确需要方案作者回答的问题
- 优先级：P0（阻塞后续设计）/P1（影响实现）/P2（优化建议）
```

### 应用场景

- 技术选型未说明理由
- 数据流向不清晰
- 关键接口未定义
- 性能指标不合理
- 依赖关系不明确
- 逻辑矛盾或不一致
- 缺少错误处理
- 并发场景未考虑
- 扩展性方案缺失
- 监控指标不完整

### 使用示例

```javascript
{
  id: 'qc1',
  title: '双库同步一致性保障机制',
  desc: '方案中提到双写事务，但未详细说明当向量库写入失败时的回滚机制，以及如何保证双库数据的最终一致性。',
  priority: 'P0',
  status: 'questioning',  // 特殊状态：待澄清
  category: '疑问澄清',
  type: 'clarification',  // 新增类型
  checklist: [
    { id: 'c1', text: '向量库写入失败的回滚流程是否已设计？', checked: false },
    { id: 'c2', text: '是否有定时一致性校验机制？', checked: false },
    { id: 'c3', text: '双库数据不一致时的自动修复方案？', checked: false },
    { id: 'c4', text: '需要补充：双写事务的详细流程图', checked: false },
    { id: 'c5', text: '需要补充：异常场景的处理策略', checked: false }
  ],
  question_type: 'missing_detail',  // 问题类型
  impact_area: ['数据入库', '系统稳定性'],  // 影响范围
  suggestions: [
    '参考 Saga 模式设计分布式事务',
    '增加定时全量一致性校验任务',
    '设计数据对比与自动修复工具'
  ]
}
```

### 问题类型分类

| 类型 | 说明 | 示例 |
|------|------|------|
| `missing_detail` | 缺少关键细节 | 未说明错误处理流程 |
| `logic_conflict` | 逻辑矛盾或冲突 | 性能指标与架构设计不符 |
| `unclear_requirement` | 需求不明确 | "合理的响应时间"无具体数值 |
| `tech_uncertainty` | 技术方案不确定 | 未说明为何选择该技术栈 |
| `dependency_missing` | 缺少依赖说明 | 未说明依赖的外部系统 |
| `edge_case` | 边界情况未考虑 | 未说明数据量超预期的处理 |
